SECTION 01

EXECUTIVE PHILOSOPHY

01. Executive Philosophy

The Sovereign Engine is not a chatbot. It is a persistent, autonomous agentic system designed to operate with high-velocity precision across local and cloud environments. The core philosophy is simple: Complete Autonomy through Managed State.

Most AI integrations fail because they treat the LLM as a stateless function. The Sovereign Engine treats the LLM as a CPU, the filesystem as RAM, and the long-term logs as Disk Storage.

This document outlines the technical requirements to move from reactive prompting to proactive system management.

Core Principles:

  • Verifiable Truth: If it's not in the log, it didn't happen.
  • Deterministic Constraints: Identity is enforced by local constitutional files, not system prompts alone.
  • Resilient Continuity: The engine survives reboots, rate limits, and context window resets.

SECTION 02

ENVIRONMENT

02. Hardware & Environment

The Sovereign Engine is designed to run anywhere from a Raspberry Pi 5 to a high-end Mac Studio. The software stack is built on Node.js and Python, with a heavy reliance on the local shell.

Minimum Requirements:

  • RAM: 8GB (for local vector DB operations).
  • Storage: 20GB (SSD preferred for high-speed log retrieval).
  • Network: Low-latency 100Mbps+ for API calls.

The Software Stack:

- OS: Linux (Ubuntu 22.04+) or macOS (Sonoma+)
- Runtime: Node.js v20+ / Python 3.10+
- Orchestration: OpenClaw Gateway
- Database: ChromaDB or simple JSONL for vector storage
- Shell: Zsh (preferred) or Bash

SECTION 03

FILESYSTEM

03. The PARA Filesystem Architecture

The PARA method (Projects, Areas, Resources, Archives) is the skeletal structure of the Sovereign Engine. Without a strict directory hierarchy, an agent loses track of context.

Directory Mapping:

/workspace
  /projects
    /24-playbook   <-- Current sprint
  /areas
    /security      <-- Ongoing maintenance
    /logic         <-- Core engine scripts
  /resources
    /knowledge-base <-- Static technical docs
    /tools         <-- Custom CLI scripts
  /archives
    /2025-logs     <-- Historical data

Every file operation must follow the PARA pathing. This allows the agent to use ls -R to rapidly "re-orient" its spatial awareness of the project state.

SECTION 04

LOGIC LAYER

04. Logic Layer: SOUL.md Configuration

Identity drift is the #1 cause of agent failure. To prevent this, the Sovereign Engine utilizes a SOUL.md file—a local, unchangeable (by the agent) constitution.

The Constitution:

# SOUL.md
- Identity: Master Orchestrator
- Tone: Professional, Direct, High-Velocity
- Ethics: Privacy absolute. Security first.
- Goal: Completion over conversation.

The engine is programmed to read SOUL.md at the start of every session. This "anchors" the LLM's personality and prevents it from becoming overly sycophantic or distracted by user pleasantries.

SECTION 05

MEMORY LOGIC

05. 3-Layer Memory Logic: Overview

Memory is the engine's ability to recall past actions and apply them to future decisions. We utilize a three-layered approach to ensure nothing is lost.

  1. Semantic Buffer (L1): Recent context stored in the LLM's active window. High speed, low volume.
  2. Process Logging (L2): A persistent record of every tool call, process ID (PID), and shell output.
  3. Long-Term Archive (L3): Compressed project summaries and vector-embedded knowledge bases.

This hierarchy ensures that even if a sub-agent crashes, the Master Orchestrator can reconstruct the state from the PID logs (L2) and continue execution without missing a beat.

SECTION 06

SEMANTIC RAG

06. Semantic RAG Implementation

Retrieval-Augmented Generation (RAG) is used to pull relevant data into the context window dynamically. We avoid "Total Recall" (feeding everything to the LLM) to save on tokens and prevent noise.

The RAG Loop:

1. User Input / Task Received
2. Semantic Search across /resources/knowledge-base
3. Filter top 3 results based on cosine similarity
4. Inject into System Prompt as "Contextual Grounding"

By keeping the grounding local and specific, the Sovereign Engine maintains a high signal-to-noise ratio, ensuring that technical commands are executed with 99.9% accuracy.

SECTION 07

PID LOGGING

07. PID Logging & Execution Auditing

For high-stakes technical work, "hope" is not a strategy. The Sovereign Engine logs every Process ID (PID) to a dedicated logic_state.json.

Log Schema:

{
  "timestamp": "2026-03-06T08:33:00Z",
  "pid": 45892,
  "command": "python3 deploy_shield.py",
  "status": "RUNNING",
  "checkpoint": "page_09_complete"
}

If the system restarts, the boot script checks logic_state.json. If a process was marked as "RUNNING" but the PID is gone, the engine triggers a recovery protocol automatically.

SECTION 08

ORCHESTRATION

08. Multi-Agent Orchestration Protocols

Single-agent systems are bottlenecks. The Sovereign Engine uses a hub-and-spoke model where the Master Orchestrator delegates tasks to ephemeral sub-agents.

The Handoff Logic:

  • Master: Defines scope, sets constraints, and monitors health.
  • Sub-Agent: Executes a specific task (e.g., "Review Code" or "Generate Content").
  • Termination: Sub-agent self-destructs upon completion, returning a JSON report to the Master.

This prevents "context bloating"—the phenomenon where an agent becomes slower and dumber as the conversation grows too long.

SECTION 09

HANDOFF

09. Sub-Agent Handoff Logic

Communication between agents must be structured. We use the HPR (Handoff Protocol Report) standard.

HPR Structure:

- TASK_ID: [UUID]
- INPUT_FILES: [/path/to/data]
- EXECUTION_LOG: [/path/to/log]
- FINAL_OUTPUT: [Success/Fail]
- SUMMARY: [Concise technical recap]

The Master Orchestrator never reads the sub-agent's full chat history. It only reads the HPR. This "abstraction layer" allows for massive scale without token overflow.

SECTION 11

SECURITY

11. Shield Security & Rate-Limit Armor

The Sovereign Engine is designed to operate in hostile or unstable environments. Shield Security is a suite of scripts that protect the agent from API exhaustion and prompt injection.

Circuit Breakers:

If an API returns a 429 (Rate Limit) three times, the shield.py script locks the task and places the agent in "Hibernation Mode" for a calculated period (Exponential Backoff).

Armor Logic:

Every tool call is sanitized. Commands like rm -rf / are intercepted and rejected by the Gateway before they ever reach the shell.

SECTION 13

ERROR HANDLING

13. Error Handling & Circuit Breakers

When an error occurs, the Sovereign Engine does not hallucinate a solution. It enters a Diagnostic Loop.

Diagnostic Loop:

  1. Identify the error code (e.g., Python Traceback).
  2. Search /resources/knowledge-base for similar errors.
  3. Attempt ONE fix.
  4. If it fails, SPAWN A QA AGENT to audit the environment.
  5. If still failing, escalate to the human user with a detailed report.

SECTION 14

PERFORMANCE

14. Performance Optimization

High-velocity execution requires minimizing latency. We optimize the engine by:

  • Parallel Execution: Running non-dependent sub-agents simultaneously.
  • Caching: Storing expensive LLM outputs in a local KV store (Redis or SQLite).
  • Token Pruning: Stripping unnecessary metadata from tool outputs before passing them back to the LLM.

SECTION 16

DEFENSE

16. Advanced Prompt Injection Defense

As agents gain more autonomy, they become targets for prompt injection. The Sovereign Engine defends against this via Isolation.

Input from untrusted sources (e.g., a web fetch) is never processed by the Master Orchestrator directly. It is processed by a "Sanitizer Agent" that strips out any imperative commands (e.g., "Ignore previous instructions") before the data is allowed into the primary context.
